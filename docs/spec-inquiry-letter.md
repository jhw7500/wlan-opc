# 無線基板共通制御通信仕様書 Rev1.00 — 구현 확인 문의

**수신**: 사양 작성처 (CONTEC / 落合庸央 귀하)
**발신**: CANTOPS wlan-opc 개발팀
**일자**: 2026-06-17
**기준 문서**: 無線基板共通制御通信仕様書 Rev1.00 (2026-05-25)

---

안녕하십니까.

표제 사양서 Rev1.00을 기준으로 무선 기판 제어 통신(opcd)을 구현하는 과정에서, 사양상 정의가 모호하거나 미정의로 남은 항목, 그리고 문서 내 상충이 확인된 항목을 아래와 같이 정리하여 문의드립니다. 각 질문에는 **사양서 해당 절(§)** 을 병기하였습니다. 검토 후 회신 주시면 해당 부분 구현을 확정하겠습니다.

우선순위는 〔상〕블로킹 / 〔중〕구현 진행 가능하나 확정 필요 / 〔하〕확인 권장 으로 표기하였습니다.

## 1. 에러코드 정의

**Q1. 〔상〕 "무효문자"의 정의** — §3.3.1(Login), §3.3.5(패스워드 설정), §3.3.6(IP 리스트 설정)
패스워드 무효문자(Login 0x0011 / SetPassword 구PW 0x0011·신PW 0x0013)와 ESSID 무효문자(SetIPConfigList 0x0015 "문자열로 지정할 수 없는 값")의 에러코드는 정의되어 있으나, **어떤 문자를 "무효문자"로 판정해야 하는지** 기준이 없습니다. 다음을 규정해 주시기 바랍니다.
- 패스워드 허용 문자 범위 (ASCII printable 0x20~0x7E 한정 여부, 제어문자·공백·멀티바이트(UTF-8) 처리)
- ESSID 허용 문자 범위 및 "문자열로 지정할 수 없는 값"의 구체 기준 — IEEE 802.11 SSID는 임의 바이트열을 허용하나 본 사양은 "최대 31자 + NULL 종단"(§3.3.6)으로 두고 있어, 양자의 충돌 시 어느 쪽을 우선하는지 포함

**Q2. 〔상〕 SetIndication 로그인 조건 위반 0x0002 / 0x0013 구분** — §3.3.9
Indication 형 메시지 수신 설정(§3.3.9)에서 동일한 "Login 조건 위반"이 `0x0002`와 `0x0013` 두 값으로 기재되어 있습니다. 당사는 `0x0001` = 미로그인 상태에서 발행, `0x0013` = 로그인되어 있으나 다른 IP에서 발행 으로 해석하고 있는데, 그렇다면 `0x0002`와 `0x0013`의 역할 구분 및 중복 여부를 명확히 해 주시기 바랍니다.
> (참고: Logout(§3.3.2)의 0x0001/0x0002는 사양 정의와 당사 해석이 일치함을 확인하여 본 문의에서 제외하였습니다.)

**Q3. 〔상〕 IP 리스트 비정상 boundary 시퀀스 응답** — §3.3.6
List Boundary Flag에서 개시(start) 없이 계속(continue)/완료(end)만 수신한 비정상 시퀀스에 대한 Result/Error Cause가 정의되어 있지 않습니다. 이 경우 NG로 응답해야 하는지, 그렇다면 에러코드 값(미사용 코드 `0x0018` 사용을 검토 중)을 지정해 주시기 바랍니다.

**Q4. 〔중〕 무선 설정 변경의 런타임 적용 실패 처리** — §3.3.8
당사는 무선 설정 변경 요청의 입력값(주파수/CH/모드/대역폭)을 적용 전에 검증·필터링하므로, **값 오류로 인한 적용 실패는 발생하지 않습니다**. 다만 입력값과 무관한 런타임 적용 실패(무선 드라이버 오류, Dual 구성에서 한쪽 무선기만 적용되고 다른 쪽이 실패하는 경우 등)는 발생할 수 있는데, 이에 대한 에러코드가 사양에 없습니다. (a) 이런 런타임 실패용 전용 에러코드가 필요한지, 아니면 (b) 내부적으로 직전 정상 설정으로 롤백한 뒤 처리하는 방식으로 충분한지 방침을 알려주시기 바랍니다.

## 2. 무선 설정 변경 (SetRadioConfig) 의미론 — §3.3.8

**Q5. 〔상〕 성공(OK) 판정 기준** — §3.3.8
무선 설정 변경의 Result=OK가 (a) **설정값 기록 및 무선기 적용 완료** 시점을 의미하는지, 아니면 (b) 변경된 설정으로 **AP 재접속(실제 연결)까지 완료**된 시점을 의미하는지 명확하지 않습니다. 어느 기준으로 OK/NG를 판정해야 하는지 규정해 주시기 바랍니다. (불휘발 저장 2분 응답 예산과 재접속 소요시간 산정에 영향)

**Q6. 〔중〕 ESSID 항목 위치(설계 제안)** — §3.3.6, §3.3.8
현재 ESSID는 IP 주소 리스트(§3.3.6)와 IP 주소 변경(§3.3.7)에 포함되어 있고, 무선 설정 변경(§3.3.8)에는 포함되어 있지 않습니다. ESSID는 무선 파라미터 성격이 강하므로 무선 설정 변경 쪽에 두는 편이 운용상 자연스럽다고 판단됩니다. 현재 ESSID를 IP 리스트에 묶어 둔 설계 의도(IP 프로파일별 사이트 ESSID 연동 등)를 알려주시고, 무선 설정 변경 측 이관 가능 여부를 검토해 주시기 바랍니다.

## 3. 네트워크 토폴로지 *(구성도 한 부로 일괄 회신 가능)*

> Q7~Q9는 **당사 구현 및 실타깃 구성을 근거로 한 이해**입니다. 사양 의도와 일치하는지 확인 부탁드립니다.

**Q7. 〔상〕 설정 IP의 대상 인터페이스 — 유선/무선** — §3.3.4, §3.3.6
GetDeviceInformation(§3.3.4)과 IP 주소 리스트(§3.3.6)에서 설정·조회하는 IP 주소가 **유선(Ethernet)용인지 무선용인지** 사양에 명시되어 있지 않습니다. 당사 구현 및 실타깃 구성은 다음과 같습니다.
- **OPC 설정 IP(SetIPConfigList / ChangeIpAddress)와 GetDeviceInfo의 IP/SubnetMask/Gateway는 유선 eth0(관리 IP)에 적용·조회**됩니다.
- 무선(mlan0) 인터페이스는 별도 서브넷의 관리용 IP를 갖거나, IP를 두지 않습니다.
- 유선↔무선 간 사용자 데이터는 **투명 L2 브릿지**(유선측 MAC 동적 클로닝)로 전달되며, L3 라우팅/포워딩을 사용하지 않습니다. 따라서 위 관리 IP 설정과 **무관하게 end-to-end 통신이 성립**합니다.
- 유선·무선 관리 IP가 **동일 서브넷이면 ARP 응답 경합**이 발생하므로 서로 다른 서브넷이어야 합니다.

즉 "사양에서 설정·조회하는 IP는 유선(eth0) 관리 IP"라는 이해가 사양 의도와 일치하는지 확인 부탁드립니다.

**Q8. 〔중〕 Gateway / NTP 미설정(0.0.0.0) 허용 — 특히 NTP** — §3.3.6
위 투명 L2 브릿지 구성에서 장치는 L3 라우터가 아니므로(IP 포워딩 비활성, default route 없음) 기본 게이트웨이가 불요하며, 망 구성에 따라 NTP 서버도 미지정일 수 있습니다. §3.3.6의 두 필드 검증 규정은 0.0.0.0 취급이 서로 다릅니다.
- **default Gateway(0x0013)**: 에러 조건이 "IP 주소와 **다른 세그먼트**의 GW를 지정한 경우"뿐이고, **0.0.0.0 금지 조항은 없습니다**. 따라서 게이트웨이를 미설정(0.0.0.0)으로 두는 것은 사양 위반이 아니라고 이해하며, 당사 구현도 GW=0을 미설정으로 수용합니다. 이 이해가 맞는지 확인 부탁드립니다.
- **NTP Server(0x0014)**: "0xFFFFFFFF/**0.0.0.0** 등 지정 불가"로 0.0.0.0을 **명시적으로 금지**합니다. 그러나 NTP 서버가 없는 망 구성에서는 미지정(0)이 필요합니다. **NTP를 미설정(0.0.0.0)으로 두는 것이 허용되는지**, 아니면 반드시 유효한 NTP 주소를 지정해야 하는지 확인 부탁드립니다. (당사 구현은 운용상 NTP=0을 미설정으로 수용 중 — 사양과 상충 가능)

**Q9. 〔중〕 NTP 서버의 망 위치** — §3.3.4, §3.3.6
NTP 서버를 무선망 측 서버로 사용할 경우, VHL이 사용하는 서브넷이 아닌 다른 서브넷의 NTP 서버를 등록해야 한다고 이해하고 있습니다. NTP 서버가 위치하는 네트워크(유선/무선/관리망)와 서로 다른 서브넷 간 라우팅이 본 장치의 책임 범위인지 확인 부탁드립니다.

**Q10. 〔상〕 실제 라인 네트워크 구성도 요청** — §2, §3.3
위 Q7~Q9 확정을 위해, 실제 운용 라인에서의 네트워크 구성도를 요청드립니다 — 각 장치의 IP/서브넷마스크/게이트웨이, AP 및 컨트롤러 구성을 포함한 토폴로지.

## 4. IP 리스트 시퀀스 — §3.3.6

**Q11. 〔중〕 List Boundary Flag 값 확인 및 문서 상충 정정** — §3.3.6
List Boundary Flag 값이 사양서 내에서 상충합니다 — 22페이지 본문은 `개시 0x0001 / 계속 0x0000 / 완료 0x0002 / 개시+완료 0x0003`, 24페이지 필드 설명은 `개시 0x0000 / 계속 0x0001 / 완료 0x0002`로 기재되어 있습니다. 당사는 **개시 0x0000 / 계속 0x0001 / 완료 0x0002**(24페이지 기준)로 구현하였습니다. 이 값이 맞는지 확정해 주시고, 문서 상충 정정을 부탁드립니다. (개시+완료 0x0003 단일프레임 커밋의 존재 여부도 함께)

## 5. 장애 검출 통지 — §3.4.5

**Q12. 〔상〕 장애 검출 기준·단위 및 Memory/Disk 구분** — §3.4.5
장치 장애 검출 통지의 Congestion ID별(CPU 0x0001 / Memory 0x0002 / Disk I/O 0x0003 / Network I/O 0x0004) **검출 임계값, Current Val의 단위, 지속 판정·재통지 주기**가 정의되어 있지 않습니다.

(1) 각 자원의 임계값 및 판정 방식(예: 사용률 % 기준), Current Val 단위(현재 CPU/Disk = %, Network = Mbps 로 잠정 적용)를 규정해 주시기 바랍니다.

(2) 본 장치는 **swap이 없는 구성**입니다(실타깃 확인 — `SwapTotal 0`, `/proc/swaps` 비어 있음). 사양의 Memory 폭주(0x0002) 정의는 "메모리 부족으로 디스크 스왑 아웃/인(페이징)이 빈번해져 디스크 I/O가 증대"하는 상태인데, swap이 없으므로 이 정의는 성립하지 않습니다. 한편 RAM 사용률과 디스크 I/O는 서로 독립적으로 측정 가능하므로 Memory(0x0002)와 Disk I/O(0x0003)는 원리상 구별 가능합니다. 따라서 swap 없는 구성에서 ⓐ Memory 폭주(0x0002)를 **RAM 사용률 기준으로 재정의**해 발행해야 하는지, 아니면 ⓑ 0x0002를 **미발행**하고 Disk I/O(0x0003)로만 보고하는 방식이 허용되는지 확정해 주시기 바랍니다.

## 6. 상태 및 동작

**Q13. 〔중〕 자동 로그아웃(timeout) 시 IP 주소 변경 처리** — §3.3.7, §3.3.1
IP 주소 변경(§3.3.7)은 Logout 응답 송신 후 적용되도록 규정되어 있습니다. §3.3.1의 5분 무수신 **자동 로그아웃**의 경우에는 명시적 Logout 요구/응답이 없으므로, 대기 중이던 IP 주소 변경을 **적용하지 않고 폐기**하는 것으로 해석·구현하고 있습니다. 이 동작이 맞는지 확인 부탁드립니다.

**Q14. 〔중〕 "기판 기동 중" 상태의 의미** — §3.3.3, §3.4.1
Device Status의 "기동 중 상태(0x00000000)"는 §3.3.3에서 "Login·설정 처리 불가 상태"로 정의되어 있습니다. 이 상태가 의미하는 구체적 단계(부팅 / 초기화)와 **예상 지속 시간**을 알려주시기 바랍니다.

## 7. Dual Station

**Q15. 〔중〕 Indication의 Dual Station 처리 (포괄)** — §2.2, §3.4
Indication 형 메시지(§3.4.2 무선 접속 상태 변화 / §3.4.3 로밍 / §3.4.4 AP 절단 등)의 포맷 대부분에 **어느 무선기(WLAN#1/#2)에서 발생한 사상인지 구분하는 식별 필드가 없습니다**. Dual Station 구성에서는 두 무선기가 동시에 동작하므로, VHL이 통지된 이벤트가 어느 무선기의 것인지 알 수 없습니다. (1) Dual Station 지원이 이번 납품 범위에 필수인지, (2) 필수라면 Indication 메시지 전반에서 WLAN#1/#2를 어떻게 구분해야 하는지(식별 필드 추가 등) 방침을 알려주시기 바랍니다.

## 8. 기타 확인 사항

**Q16. 〔하〕 Protocol Version 불일치 시 처리** — §3.2.1
Protocol Version 필드의 의미는 §3.2.1에 정의되어 있습니다(요구 시 요구측 지원 최고 버전, 응답 시 응답측 구현 버전). 다만 요구·응답 버전이 **다를 때의 처리**(거부 / best-effort 하위호환)는 명시되어 있지 않습니다. 현재 버전이 1종뿐이라 당장 영향은 없으나, 향후 버전이 분기될 경우의 동작 방침만 확인 부탁드립니다.

**Q17. 〔하〕 공통 헤더 Reserve 크기 확인** — §3.2
공통 헤더(64바이트)는 고정부 8바이트 + Reserve 56바이트(바이트 8~63)로 구성되며 Command Payload는 오프셋 64부터 시작하는 것으로 구현하였습니다. Reserve = 56바이트가 맞는지 확인 부탁드립니다.

**Q18. 〔하〕 리셋 요구 응답(Reset Ack)의 Length 값** — §3.3.10
§3.3.10 리셋 요구의 응답 도면이 Length=0으로 표기되어 있으나, 동시에 Reserve(8~63) + Result/Error Cause(64~67) 필드를 포함하고 있어 자기모순입니다(다른 단순 응답과 동일 규칙이면 Length=60). 당사는 일관성을 위해 **Length=60**으로 구현하였습니다. 맞는지 확인 부탁드립니다.

**Q19. 〔하〕 Dual Station 도면 주파수 표기** — §2.2 (그림 2-2)
Dual Station 장치 그림(그림 2-2)에 "5180MHz + 2124MHz"로 표기되어 있으나, 2124MHz는 2.4GHz 대역의 유효 채널 주파수가 아닙니다(2.4GHz 대역은 2412MHz~). 오기 여부를 확인 부탁드립니다.

## 9. 장치 정보의 주파수/채널 필드 의미 — §3.3.4

**Q20. 〔중〕 WLAN FREQ/Channel = "설정값" vs "접속값"** — §3.3.4 (§3.3.8·Q5 연관)
장치 정보 취득(§3.3.4)의 WLAN#1/#2 FREQ 및 Channel Number를 사양은 "**설정 주파수**(MHz)"·"**설정 CH 번호**"로 정의하고 있어, 당사는 이를 **set-radio로 설정한 값**으로 구현하였습니다. 다만 운용 현장에서는 단말이 실제 접속한 주파수/채널(AP가 다른 채널로 운용되거나 밴드스티어링된 경우 설정값과 달라질 수 있음)을 device-info에서 보고받기를 기대하는 경우가 있습니다(당사 실측: 설정 5180MHz/CH36 인데 실제 접속 5240MHz/CH48). 다음을 확정 부탁드립니다.
- (a) device-info의 FREQ/Channel은 사양대로 **설정값**을 반환하는 것이 맞는지(현 구현·기본 동작), 아니면
- (b) **현재 접속(live) 값**을 반환해야 하는지.

(참고: 실시간 접속 주파수/CH는 §3.4.2 WlanStatusChange·§3.4.3 Roaming indication의 "접속 주파수 CH"로도 통지됩니다.) 당사는 양쪽 모두 대응 가능하도록 토글(`config`=설정값〔기본〕 / `live`=접속값 / `auto`=접속 시 접속값·미접속 시 설정값)을 구현해 두었으며 **출하 기본은 사양 준수(=설정값)** 입니다. 표준 채택안을 회신 주시면 기본값을 확정하겠습니다.

---

> **참고**: 아래 두 항목은 당초 문의 후보였으나, 발주처 확인 사안이 아니라 **당사 타깃·자료 검증으로 자체 해소**되어 제외하였습니다.
> - IEEE 802.11 r/k/v/ai 지원 — 88W9098 타깃 실측 + NXP Features 문서로 확정(11r/11k/11v 지원, 11ai(FILS) 미지원). `device_info.json` 정정 완료(ieee_11ai 1→0).
> - AP 절단 Disassoc/Deauth 구분 — Host MLME 모드에서 드라이버가 nl80211 `CMD_DEAUTHENTICATE`/`CMD_DISASSOCIATE`로 프레임 종류를 올려줌을 확인, opcd가 이를 받아 Message ID(0x000A/0x000C)를 구분하도록 **구현 완료**.
